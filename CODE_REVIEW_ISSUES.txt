[HIGH] [FIXED] ISSUE-006: getUserId() silently returns null; unauthenticated writes are allowed
File : src/Controller/Api/TodoController.php
Fix  : All mutation endpoints now call denyAccessUnlessGranted()
       (TodoPermission::Manage) before reaching getUserId(), so the method

[HIGH] [OPEN] ISSUE-007: Pimcore Studio UI frontend integration is missing
Files : assets/studio/js/src/index.tsx
        assets/studio/js/src/modules/todo-bundle/components/Dashboard.tsx
Detail: The index.tsx entry point mounts the React app by looking for a raw
  DOM element with id="studio-todo-root". Pimcore Studio UI uses a module/
  widget registration API (typically via @pimcore-studio/ui or the Pimcore
  Studio SDK). The bundle does not register itself as a Studio widget, does not
  integrate with Studio's navigation, and does not use Studio's routing system.
  The bundle will therefore not appear in Pimcore Studio UI.

[HIGH] [OPEN] ISSUE-008: No CSRF protection or Pimcore auth token in frontend API calls
File : assets/studio/js/src/modules/todo-bundle/services/todoApi.ts
Detail: The Axios instance is created with only Content-Type headers. Pimcore
  Studio UI API calls require the Pimcore auth token (X-Pimcore-Simple-Auth-
  Token or cookie-based session). Without this, all API calls will be rejected
  by Pimcore's authentication layer once ISSUE-002 is fixed.

[HIGH] [OPEN] ISSUE-009: Missing Pimcore 12 bundle metadata / pimcore.yml / attributes
File : src/StudioTodoBundle.php
Detail: Pimcore 12 recommends bundles declare themselves via the
  #[AsPimcoreBundle] attribute (or equivalent configuration) for auto-
  discovery features. The extra.pimcore.bundles key in composer.json is the
  legacy Pimcore auto-registration mechanism and may not work reliably in
  Pimcore 12 depending on the project setup.

[HIGH] [FIXED] ISSUE-010: Double event dispatch on complete() – UPDATED fired twice
File : src/Service/TodoManager.php
Fix  : complete() no longer delegates to update() (which would dispatch
       TodoEvents::UPDATED). Instead it calls $this->repository->update()
       directly and dispatches only TodoEvents::COMPLETED. This eliminates
       the duplicate UPDATED event for listeners.

[HIGH] [FIXED] ISSUE-011: StatisticsService::getStatisticsByCategory() loads up to 10 000 rows in PHP
File : src/Service/StatisticsService.php  (replaced)
       src/Repository/TodoRepository.php  (added getStatisticsByCategory())
Fix  : Added getStatisticsByCategory() to TodoRepository using a SQL
       GROUP BY query (COALESCE(category, 'uncategorized')) matching the
       pattern already used by getStatisticsByUser(). StatisticsService
       now calls this repository method instead of loading 10 000 rows.

[MEDIUM] ISSUE-012: TodoManager::cleanup() loads all completed todos into memory
File : src/Service/TodoManager.php  lines 278-296
Detail: cleanup() calls findAll() with a status filter but no date condition,
  loading every completed todo into memory and filtering by completedAt in
  PHP. With large datasets this will exhaust memory. A targeted database query
  (WHERE completed_at < :cutoff) should be used instead.

[MEDIUM] ISSUE-013: CleanupCommand fetches all todos including deleted for hard-delete scan
File : src/Command/CleanupCommand.php  lines 77-89
Detail: The command calls findAll(['include_deleted' => true]) with no further
  filter, loading every todo record (including non-deleted ones) to find those
  whose deletedAt is older than the cutoff. A targeted SQL query is required.

[MEDIUM] ISSUE-014: bulkUpdate() and bulkDelete() execute N individual queries
File : src/Service/TodoManager.php  lines 302-325
Detail: Both methods loop over IDs and call update()/softDelete() for each one
  individually. Each call involves a SELECT + UPDATE round-trip, plus event
  dispatch and async message per item. For large bulk operations this is very
  slow. A single batch UPDATE query with an IN clause should be used.

[CRITICAL] [FIXED] ISSUE-015: No input validation on JSON body in create/update endpoints
Files : src/Controller/Api/TodoController.php
Fix  : Added is_array() check immediately after json_decode(). If the body
       is not valid JSON or decodes to a non-array, a 400 Bad Request
       response is returned before any further processing. Applied to
       create(), update(), bulkUpdate(), and bulkDelete().

[HIGH] [FIXED] ISSUE-016: Bulk operation IDs are not cast to integers before database use
File : src/Controller/Api/TodoController.php
Fix  : In bulkUpdate() and bulkDelete(), the $data['ids'] array is now
       filtered through array_map('intval') followed by array_filter()
       to ensure only positive integers are passed to the service layer.

[HIGH] [FIXED] ISSUE-017: Arbitrary search string is concatenated before binding
File : src/Repository/TodoRepository.php
Fix  : Added substr() truncation to SEARCH_MAX_LENGTH (200 chars) in
       createFilteredQuery() before building the LIKE parameter. The value
       is still bound as a prepared-statement parameter (safe from SQL
       injection), and DoS via extremely long LIKE scans is now mitigated.

[MEDIUM] [OPEN] ISSUE-018: Date strings are parsed with new \DateTimeImmutable() without try/catch
Files : src/Model/TodoItem.php  lines 116-120
        src/Repository/TodoRepository.php  line 380
Detail: Malformed date strings in database rows or API input will throw an
  unhandled \Exception. The fromArray() method and formatDateTime() method
  should catch \Exception and either return null or throw a domain-level
  exception with a meaningful message.

[MEDIUM] [OPEN] ISSUE-019: TodoOperationMessage carries raw $data array from user input
File : src/Message/TodoOperationMessage.php
Detail: The $data property is populated with the raw parsed request body from
  the controller before any validation. Sensitive fields (e.g. passwords if
  added in meta) or excessively large payloads travel through the message bus
  and are potentially stored in the queue backend. Only the minimal data
  required for async processing should be carried in the message.
